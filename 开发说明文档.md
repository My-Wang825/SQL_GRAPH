# SQL 知识图谱系统 — 开发说明文档

本文档按**运行与数据处理的先后顺序**，梳理从环境准备到前端展示的完整流程，并说明与指标血缘、智能问答等扩展能力的关系。

---

## 一、系统目标与数据形态

1. **目标**：从数仓 ETL 类 SQL 脚本中自动解析表名与**显式数据流**关系（`INSERT INTO 目标表 … SELECT … FROM/JOIN 源表`），构建表级知识图谱，并通过 Web 力导向图展示、筛选与查询。
2. **输入数据**：项目根目录下 `data/` 中的业务板块子目录（如 `PRD_AL` 电解铝、`PRD_AO` 氧化铝），内含 `.sql` / `.txt` 脚本；表注释可来自文件头「描述：」类注释。
3. **输出形态**：内存中构建 JSON（`nodes` + `links` + `meta`），经 FastAPI 暴露为 `/api/graph`，由前端 D3 拉取并渲染。

## 二、环境与依赖（流程第 0 步）

| 顺序 | 步骤 | 说明 |
|------|------|------|
| 1 | 安装 Python | 建议 **Python 3.8+**。 |
| 2 | 安装依赖 | 在项目根目录执行 `pip install -r requirements.txt`（含 FastAPI、Uvicorn、Pydantic、OpenAI 兼容客户端、python-dotenv、openpyxl 等）。 |
| 3 | 可选环境变量 | 复制 `.env.example` 为 `.env` 并按需填写（**勿将含真实密钥的 `.env` 提交仓库**）。详见下文「扩展能力所需配置」。 |
| 4 | 浏览器与网络 | 前端通过 CDN 加载 D3.js、marked、DOMPurify；需能访问对应 CDN，或自行改为本地静态资源。 |

## 三、服务启动流程

| 顺序 | 步骤 | 说明 |
|------|------|------|
| 1 | 进入项目根目录 | 根目录须同时包含 `backend/`、`frontend/`、`data/`，否则静态资源或数据路径会错误。 |
| 2 | 启动 Web 服务 | 推荐：`python run_server.py`（默认绑定 `127.0.0.1`，端口优先 8000，占用时自动尝试其它端口）。开发热更新：`python run_server.py --reload`。 |
| 3 | `run_server.py` 行为 | 解析 `--host` / `--port` / `--reload`；检测端口可用性并可能自动换端口；调用 `uvicorn.run("backend.main:app", …)`。 |
| 4 | `backend/main.py` 初始化 | 创建 FastAPI 应用；挂载 CORS；`include_router` 注册 `backend/api/routes.py` 中的 API；定义 `/health`、显式 `/` 返回 `frontend/index.html`；最后将 `frontend/` 挂载为静态目录（供 `css/`、`js/`、`ao.html` 等访问）。 |

**访问入口**：

- 电解铝默认页：`http://127.0.0.1:<端口>/` → `index.html`，`body` 上 `data-sector="PRD_AL"`。
- 氧化铝：`http://127.0.0.1:<端口>/ao.html`，`data-sector="PRD_AO"`。
- 健康检查：`/health` → `{"status":"ok"}`。

## 四、图谱构建主流程（后端核心链路）

以下顺序对应 `build_graph_payload`（`backend/api/routes.py`）在响应 `GET /api/graph` 或 `POST /api/graph/build`、以及 Agent 复用图谱时的逻辑。

### 4.1 解析数据目录

| 顺序 | 步骤 | 说明 |
|------|------|------|
| 1 | `_resolve_graph_dir(sector, data_dir)` | **若带 `sector`**：必须在 `backend/config.py` 的 `SECTOR_DATA_DIRS` 白名单中（当前为 `PRD_AL`、`PRD_AO`），返回对应绝对路径。**若带 `data_dir`**：解析为项目根下的相对或绝对路径，且**必须位于项目根目录内**（安全限制）。**若两者皆空**：默认使用 `PRD_AL`。 |
| 2 | 校验目录存在 | 不存在则返回 HTTP 400。 |

### 4.2 缓存（可选）

| 顺序 | 步骤 | 说明 |
|------|------|------|
| 1 | 缓存键 | `sector|数据目录绝对路径`（sector 大写规范化）。 |
| 2 | TTL | 默认 120 秒内重复请求可直接返回缓存的 JSON（`use_cache=True` 时；`POST /api/graph/build` 强制 `use_cache=False` 全量重算）。 |

### 4.3 扫描 SQL 文件

| 顺序 | 步骤 | 说明 |
|------|------|------|
| 1 | `_collect_sql_files(dir_to_use)` | 递归遍历目录，收集扩展名为 `backend/config.py` 中 `SQL_EXTENSIONS`（`.sql`、`.txt`）的文件列表。 |

### 4.4 逐文件解析（ETL SQL → 表与关系）

| 顺序 | 步骤 | 说明 |
|------|------|------|
| 1 | 实例化 | `ETLSqlParser()`（`backend/parser/etl_sql_parser.py`）。 |
| 2 | 对每个文件 | `parser.parse_file(fp)` → `(table_list, relations, …)`。单文件失败时记录 warning 并跳过，不中断全库。 |
| 3 | 解析器职责（概要） | 识别 `INSERT INTO`、`TRUNCATE`、`DELETE FROM`、`FROM`/`JOIN` 中的物理表；从 `INSERT…SELECT…` 上下文提取 **DATA_FLOW**：源表 → 目标表；规范化表名（去 schema、短表名）；从文件头注释提取「描述」作为表注释。 |
| 4 | 写入构建器 | `GraphBuilder.add_tables_from_parse`、`add_relations_from_parse`。 |

**说明**：`backend/analyzer/relation_analyzer.py` 提供另一套「合并 + 传递闭包标记」逻辑；当前主路径的合并与去重在 `GraphBuilder.to_vis_json` 中完成，`RelationAnalyzer` 是否被其它脚本单独使用以仓库内引用为准。

### 4.5 图谱合并与 JSON 输出（GraphBuilder）

| 顺序 | 步骤 | 说明 |
|------|------|------|
| 1 | `to_vis_json()` | 对相同 `(source, target)` 去重，合并 `weight`、`sources` 等；为节点计算 `relationCount`；边字段映射为前端约定（如 `relationType`、`isTransitive` 固定为 `false`）。 |
| 2 | 节点类型 | `_infer_table_type` 按表名前缀启发式分类（如 `dim_`/`dm_`、`ods_`、`dwd_`/`dws_`、`ads_`、临时表、`v_` 视图等）。 |
| 3 | 输出结构 | `{ "nodes", "links", "meta" }`，供 Pydantic `GraphResponse` 校验返回。 |

### 4.6 指标血缘与治理表合并（条件执行）

| 顺序 | 步骤 | 说明 |
|------|------|------|
| 1 | `should_apply_metrics_merge_for_sector(...)` | **仅**在电解铝场景合并：`sector=PRD_AL`，或未指定 sector 但解析目录路径中含 `PRD_AL`；**`PRD_AO` 不合并**。 |
| 2 | `merge_metrics_lineage_into_payload(...)` | 读取 `METRICS_GOVERNANCE_MERGE_JSON` 指向的 JSON（默认 `exports/metrics_lineage_test/lineage_for_governance_merge.json`），将指标平台侧 **PHYSICAL_TABLE → DATASET** 与治理表节点对齐后并入图谱（含数据集虚拟节点、`METRICS_DATASET` 边或节点标注 `metricsSameAsDataset` 等）。详见 `backend/graph/metrics_governance_merge.py` 头部注释。 |

### 4.7 返回 API / 写入缓存

| 顺序 | 步骤 | 说明 |
|------|------|------|
| 1 | `GET /api/graph` | `sector`、`data_dir` 查询参数传入上述流程，返回 `GraphResponse`。 |
| 2 | `POST /api/graph/build` | 请求体中的 `data_dir` 参与解析，且不使用缓存。 |

## 五、前端展示流程

| 顺序 | 步骤 | 说明 |
|------|------|------|
| 1 | 页面加载 | `index.html` 或 `ao.html` 引入 `css/styles.css`、`js/app.js` 及 CDN 上的 D3、marked、DOMPurify。 |
| 2 | 板块 | `document.body.dataset.sector` → 常量 `SECTOR`（默认 `PRD_AL`）。 |
| 3 | 拉取数据 | `fetch("/api/graph?sector=" + encodeURIComponent(SECTOR))`。 |
| 4 | 渲染 | 使用 D3 力导向布局；按数仓分层与连通分量分槽布局；层级勾选过滤；搜索、详情、血缘高亮等（详见 `README.md` 图谱展示方案一节）。 |
| 5 | 智能问答（若启用） | 向 `/api/agent/ask` 或 `/api/agent/ask/stream` 提交问题，请求体含 `sector`；后端再次 `build_graph_payload` 取当前图谱并调用模型客户端（需 API 密钥等配置）。 |

## 六、其它 API（与主图谱链并行）

| 接口 | 作用 |
|------|------|

## 七、配置项速查（`backend/config.py` + 环境变量）

- **路径类**：`BASE_DIR`、`DATA_DIR`、`SECTOR_DATA_DIRS`、`SQL_EXTENSIONS`、`PARSER_CONFIG`、`GRAPH_VERSION`。
- **模型 / Agent**：`OPENAI_*` 或 `MINIMAX_*` 命名的 API Key、Base URL、Model（用于 OpenAI 兼容接口）。
- **指标平台**：`METRICS_LINEAGE_*`、`METRICS_TENANT_ID`、`METRICS_AUTH_*`、`METRICS_HTTP_TIMEOUT_SEC`、`METRICS_MATERIALIZED_HINTS`、`METRICS_GOVERNANCE_MERGE_JSON`。

## 八、端到端顺序小结（一张清单）

1. 准备 Python 与 `requirements.txt` 依赖。  
2. 准备 `data/` 下各板块 SQL 数据（及可选的指标合并 JSON）。  
3. 可选：配置 `.env`。  
4. 在**项目根目录**执行 `python run_server.py`。  
5. FastAPI 挂载路由与静态前端。  
6. 用户打开 `/` 或 `/ao.html` → 前端请求 `/api/graph`。  
7. 后端解析目录 → 扫 SQL → `ETLSqlParser` → `GraphBuilder` →（条件）指标合并 → 返回 JSON。  
8. 前端解析 JSON → D3 布局与交互展示。

## 附录：若仅持有本《开发说明文档.md》，还缺什么才能复刻「与本项目一致」的系统？

除本文档外，要得到**行为一致**的可运行副本，通常还需要下列材料或等价物（按重要性排列）：

1. **完整源代码**：`backend/`（含 `parser`、`analyzer`、`graph`、`api`、`models`、`metrics`、`agent` 等子包）、`frontend/`（`index.html`、`ao.html`、`css/`、`js/`）、根目录 `run_server.py`。  
2. **Python 依赖清单**：`requirements.txt`（及实际安装时匹配的版本环境）。  
3. **SQL 数据集**：`data/PRD_AL`、`data/PRD_AO`（或与 `config.SECTOR_DATA_DIRS` 一致的目录结构与脚本内容）；无数据则图谱为空。  
4. **需求/设计对照**（建议保留）：`sql-knowledge-graph-prompt.md`、`README.md`，便于核对解析规则与展示约定是否与最初设计一致。  
5. **指标合并产物（若需与当前电解铝合并行为一致）**：默认路径下的 `exports/metrics_lineage_test/lineage_for_governance_merge.json`，或通过 `METRICS_GOVERNANCE_MERGE_JSON` 指向等价文件；不需要合并功能时可不配，但 PRD_AL 图将缺少指标侧节点与边。  
6. **环境变量 / `.env`**：  
   - 使用**智能问答**、Agent 流式接口：需配置 OpenAI 兼容的 `OPENAI_API_KEY`（或 `MINIMAX_*`）、`OPENAI_BASE_URL`、`OPENAI_MODEL` 等。  
   - 使用 **`/api/metrics/lineage/summary`**：需配置指标平台 URL 与租户、鉴权等。  
   - 不配密钥时，图谱主流程仍可运行，仅相关接口不可用或报错。  
7. **运行环境**：与本机/服务器一致的 Python 大版本；浏览器；若依赖 CDN，需网络可达或改为本地化 JS。  
8. **（可选）导出与脚本**：仓库中 `exports/` 下其它分析产物、一次性脚本等，不参与核心启动则非必需，但若用于复现某次报表或血缘导出，则需一并保留说明。  

**说明**：`.env.example` 仅为模板；真实密钥与内网 URL 不应依赖仓库中的占位内容，应在目标环境单独配置。
